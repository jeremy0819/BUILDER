# PART A · 給程式設計師（Engineering）

## A1. 最優先：M5.5 傳動軸（THE DRIVETRAIN）

### A1.1 問題定義
引擎（Core）蓋好了、輪子（沙盤）蓋好了，**中間沒有傳動軸**。
現況因果鏈只有一條細線：`共負比 → 難度`。

### A1.2 目標資料流（要蓋的東西）
```
STEP 2 規劃參數
  坪型組合 / 戶數 / 公設比 / 車位數 / 每坪均價 / 總銷 / 共負比 c
        ↓
§56 逐戶權變（Core，已存在）
  更新前價值ᵢ（係數表）→ 權值比例ᵢ → 分回價值ᵢ
        ↓
【新增】選配映射層（Core 層，非 UI）
  分回權狀坪ᵢ   = 分回價值ᵢ ÷ 每坪均價
  分回室內實坪ᵢ = 分回權狀坪ᵢ × (1 − 公設比)      ← 案例C（攔胡原型）那一課
  可配單元ᵢ     = 坪型組合中 ≤ 分回價值ᵢ 的單元集合
  車位滿足ᵢ     = 是否達成一戶一車位
        ↓
【新增 schema】household_outcome[]（Core 輸出）
        ↓
沙盤意願函數（遊戲層，消費 household_outcome，不自行推導坪數）
```

### A1.3 新 Schema：`household_outcome`

### A1.4 意願來源標示（架構關鍵，勿漏）★
Sandbox 與 Workspace 的「意願」**本質不同，不得共用欄位而不標來源**：
```
Sandbox（訓練）：willingness_source = "simulated"  ← 由意願函數產生
Workspace（實戰）：willingness_source = "recorded" ← 由 ConsentEvent 記錄（Workflow 事實）
```
> 缺少此標示 = 模擬值可能被誤讀為真實同意狀態，屬**嚴重架構錯誤**。

### A1.5 意願函數：分工鐵則 ★
| 面向 | 擁有者 | 可否調整 |
|---|---|---|
| **方向（sign）** | 領域真理（仲裁人） | ❌ 不可違反，見 PART C 七條斷言 |
| **幅度（magnitude）** | 遊戲設計 | ✅ 可調，存 config 不 hardcode |

**實作要求**：意願函數係數一律存 `willingness_config.json`，遊戲設計師可調數值而**不需改程式碼**。

## A2. 實務要素資料結構（同業要求「碰實務」，只做這三個）

| 要素 | 欄位 | 效果 | 理由 |
|---|---|---|---|
| **產權複雜度** | `ownership_complexity`: `clean` / `inherited_unregistered` / `joint_ownership` / `mortgaged` / `illegal_structure` | 產生 `signable: false` — **願意但簽不了**，需額外行動與時間解鎖 | 公同共有人的真實情境；擬真度暴增、實作成本低 |
| **安置補貼** | `relocation_subsidy`（金額/戶/月 × 工期） | 意願 ↑ **且** 成本 ↑ → 共負 ↑ → 分回 ↓ → 意願 ↓（雙向張力） | 四年工期人住哪，實務天天處理 |
| **選配衝突** | `unit_inventory[]` + `selection_order` | 兩戶搶同一單元 → 需協調或抽籤 | 真實且天然有戲劇性 |

> ⚠️ 三者皆須**先進 schema 再進畫面**（紅線 3）。

# PART B · 給遊戲設計師（Game Design）

## B1. 你要修的核心體驗
> 現況：玩家在 STEP 2 認真規劃 → 進 STEP 3 發現「怎麼改都一樣」→ **規劃變成假的**。
>
> 目標：**遊戲的核心問題，從「怎麼說服人」變成「你能不能設計一個讓大家都願意的產品」。**
> 那才是整合人真正的工作，而且市面上沒有任何工具在做這件事——**這就是護城河。**

## B2. 讓玩家「感覺到」傳動軸（體感設計）

傳動軸接上後，這些操作必須**立刻、可見地**改變人心：

| 玩家操作 | 應有反應 | 對應真實課題 |
|---|---|---|
| 公設比 33% → 38% | 全體 **室內實坪縮水** → 意願全面下滑 | 案例C原型「1坪換1坪+車位」的煙霧彈 |
| 改坪型組合 | 部分戶**配不到合適單元** → 該戶意願重挫 | 選配現實 |
| 車位配比不足 | 未達一戶一位的戶 → 意願 ↓ | 契約常見爭點 |
| 提高房價／爭取容積獎勵 | 總餅變大 → 全體意願 ↑ | 防災獎勵落日的價值 |
| 加安置補貼 | 意願 ↑ **但**成本 ↑ → 分回 ↓ → 部分意願反而 ↓ | 真實兩難 |

**設計要求**：
1. **即時回饋**：規劃參數變動 → 逐戶意願條**動畫變化**，不要等結算才看到。
2. **歸因可見**：玩家點任一戶，要看到「他為什麼不爽」——`原室內 27.6 坪 → 分回 20.1 坪（−27%）`。
   **數字要說故事，不是只給結果。**
3. **蛋黃/蛋白對比**：高 V₀ 的戶（精華區）意願基準線天生較低——讓玩家親身體會 holdout 的數學根源。

## B3. 回合階段化（同業回饋：「環節跟步驟要更明確」）

M5 修好了**外層**（Site→Product→People→Decision），但**沙盤內層回合結構仍模糊**。

**建議回合結構（1 回合 = 1 季）**
```
① 情勢 Situation   本季外部事件（地價變動／競爭建商／法規時程）
② 行動 Action      花 AP 執行動作（傾聽／客製／說明會／產權清理）
③ 結算 Resolution  意願更新、同意率變化、成本累加、時間推進
④ 檢視 Review      本季發生什麼、離門檻多遠、下季提示
```

**HUD 必須永遠可見（缺一即為缺陷）**
- 第 N 回合 / 共 M 回合　·　目前階段（①②③④ 高亮）
- 剩餘 AP　·　目前同意率 vs 門檻（80% / 100%）
- 累計時間成本　·　距關鍵時程（獎勵落日等）

## B4. 三個實務要素的玩法設計

| 要素 | 玩法機制 | 設計張力 |
|---|---|---|
| **產權複雜度** | 「**願意但簽不了**」——該戶意願 100% 卻無法計入同意率，須花 AP 執行「產權清理」解鎖 | 打破「說服 = 完成」的直覺；教會玩家真實阻力常來自程序而非情緒 |
| **安置補貼** | 資源滑桿：補貼 ↑ 意願 ↑，但共負 ↑ 分回 ↓ | 沒有免費的善意——最好的兩難設計 |
| **選配衝突** | 兩戶搶同一單元 → 協調 / 抽籤 / 加碼 | 天然戲劇性；玩家理解「分配不只是數學」 |

## B5. 未來方向（**M6 之後**，現在不做）

| 項目 | 判定 | 理由 |
|---|---|---|
| **3D 量體視覺化** | ✅ 值得做 | 把 Core 的容積/樓層/戶數變成看得見的量體。是 Core 輸出的**呈現**，不是新邏輯，完全合規 |
| **數據視覺化升級** | ✅ 值得做 | 三方 EV 動態長條、逐戶意願地質圖、階段樹漏斗（案子死在哪一關） |
| **AI 談判 NPC** | ⏳ M6 之後 | 需先有 `stakeholder_profile` schema（M6）。**NPC 行為必須被 schema 綁定**，否則變成與引擎打架的失控 chatbot |
| **AI 完工渲染圖** | ❌ 先別做 | 不改變 GO/CAUTION/STOP，且可能與實際量體/法規不符而誤導 |

## B6. 設計紅線
❌ 為了好玩，讓意願方向違反領域真理（見 PART C 七條斷言）
❌ 在遊戲層自行計算坪數／財務（一律消費 `household_outcome`）
❌ 顯示任何無法溯源到 Core 的數字
❌ 把模擬意願呈現得像真實同意狀態（須標 `simulated`）

# PART C · 交叉確認機制（錯誤歸屬與修正）★

> 目的：讓「誰的錯、怎麼判、怎麼修」**可判定**，而非靠爭論。

## C1. 交接契約（Interface Contract）
```
程式交付：household_outcome[] + result JSON + willingness_config.json
遊戲消費：只讀上述輸出，產生體驗

歸屬原則：
  · 契約「之內」的數值錯誤（坪數/價值/可配單元）→ 程式
  · 契約「之外」的體驗問題（沒感覺/看不懂/不好玩）→ 遊戲設計
  · 契約本身缺欄位（想做但資料沒給）→ 先補 schema（程式），不得由遊戲層自行推算
```

## C2. 七條方向斷言（領域真理，兩方共同驗收基準）
```
D1  分回室內實坪 ↓        → 該戶意願 ↓
D2  公設比 ↑（其他不變）  → 全體意願 ↓
D3  無可配單元（配不到房）→ 該戶意願重挫
D4  車位未達一戶一位      → 該戶意願 ↓
D5  房價 ↑ / 容積獎勵 ↑   → 總餅 ↑ → 全體意願 ↑
D6  共負比 c ↑            → 分回 ↓ → 意願 ↓
D7  V₀ 高（蛋黃區）       → 意願基準線低於蛋白區
```
**程式**：把 D1–D7 寫成自動化測試。
**遊戲**：可調幅度，但**任何調整不得使 D1–D7 失效**。
**違反 = 領域錯誤，一律回退。**

## C3. 錯誤歸屬判定表
| 症狀 | 診斷方法 | 歸屬 | 修正路徑 |
|---|---|---|---|
| 遊戲顯示坪數與 Core 不符 | 比對 `result JSON` 對應欄位 | 程式 | 修傳動軸映射 |
| 改公設比，遊戲毫無反應 | 檢查 `household_outcome` 是否傳遞該欄位 | 程式 | 補欄位/接線 |
| 數值正確，但玩家「沒感覺」 | 跑敏感度：改參數後意願變化幅度 | 遊戲設計 | 調 `willingness_config` |
| 意願變化**方向**與實務相反 | 對照 C2 七條斷言 | 領域（仲裁人裁定） | 修意願規則 |
| verdict 與 Core 數字矛盾 | `input_hash` 溯源 | 程式 | Decision Engine bug |
| 玩家不知現在哪個回合階段 | 人測 | 遊戲設計 | 補 HUD（B3） |
| 畫面出現無法溯源的數字 | 跑 C4 溯源測試 | 程式 | **自動判定為程式錯誤** |

## C4. 數字溯源自動測試（最強防漂移裝置）
```
test_sandbox_numbers_traceable：
  沙盤/報告中每一個財務數字，必須對應到 result JSON 或
  household_outcome 的具體欄位路徑。
  找不到來源 → 測試失敗 → 自動歸屬程式錯誤。
```
> 這條讓「UI 零推論」從口號變成**可執行的檢查**。

## C5. 三層驗收（每次迭代都要跑）
```
第一層 · 單元（程式）：pytest / node 測試 + C2 七條斷言 + C4 溯源測試
第二層 · 整合（雙方）：三個對抗性案例 A內爆 / B背信 / C攔胡 回歸必須綠
第三層 · 人測（同業）：交付同業試玩，用 C6 回饋表收集
```

## C6. 同業人測回饋表（讓回饋可歸屬，而非模糊意見）
交付同業試玩時附此表，每則回饋填：
```
[1] 發生在哪一步驟？ Site / Product / People / Report
[2] 我做了什麼？（具體操作）
[3] 系統反應是什麼？
[4] 我期待的反應是什麼？
[5] 實務上正確的應該是什麼？（若涉領域）
[6] 嚴重度：擋路 / 困惑 / 建議
```
> 有 [4][5] 才能判定是「體驗問題」還是「領域錯誤」——這就是交叉比對的入口。

## C7. 衝突仲裁規則
```
優先序：領域真理 > 遊戲樂趣 > 實作方便

· 遊戲想要的效果違反領域真理 → 領域勝，改設計
· 遊戲想要的效果需要新領域概念 → 走 Schema First，先補 schema 再做
· 程式說「做不到」但屬契約內 → 補契約，不得叫遊戲層自行推算
· 雙方對「什麼是實務真相」有歧異 → 交仲裁人（郭哲雍）裁定，裁定寫入 lessons-learned
```

## C8. 每次迭代檢查表
```
[ ] C2 七條方向斷言全綠
[ ] C4 數字溯源測試全綠
[ ] 三個對抗性案例回歸全綠
[ ] willingness_source 標示正確（simulated / recorded）
[ ] verdict 溯源條存在且與結論同等顯眼
[ ] 無真實資料/圖檔進版控（含圖檔白名單檢查）
[ ] check_no_real_names.sh 綠
[ ] 版本號一致（首頁 / 沙盤 / report 顯示同一 core & schema 版本）
```
