# STANDARD WORKFLOW — Site → Product → People → Decision

> **層級**：最高層級產品文件（Ubiquitous Language 的唯一定義處）
> **版本**：v1.0（2026-07 擁有者定稿）
> 之後每一個新功能、每一份文件、每一次對外介紹，都從這條語言長出來。

## 一句話定義（跟外行人這樣說）

**BUILDER 是一套都市更新的決策作業系統（Decision OS）：
先看地、再定產品、然後面對人，最後告訴你——這個案子值不值得做、下一步該做什麼。**

沙盤＝訓練模式（Training），工作區＝實戰模式（Reality）——同一套 OS 的兩個模式。
它不是試算表，是把「該不該做」走成一條可稽核流程的系統。

## 為什麼是「基地 → 產品 → 人 → 決策」

因為這是**因果的順序，不是功能的順序**：

**規劃決定財務，財務決定人心。**

- 基地決定你能蓋什麼（法規與容積是天花板）；
- 產品決定財務長什麼樣（坪效、共負、分回都從產品配置長出來）；
- 財務決定每一戶拿到什麼，**拿到什麼決定他點不點頭**——人心不是玄學，是分回的函數；
- 而人能不能整合起來，才決定這個案子值不值得投。

倒過來走（先算財務、再找人背書）正是多數案子翻車的原因：
數字漂亮、人心沒過——等於零。所以順序只能是 Site → Product → People → Decision。

## 每一步在回答什麼

| STEP | 產品語言 | 問句 | 答案來源 |
|---|---|---|---|
| 1 | **Site Analysis 基地診斷** | 這塊地值不值得投？ | Core（Site facts）；「值不值得」verdict＝Decision Engine |
| 2 | **Product Planning 產品規劃** | 該蓋什麼？ | RE-DCF Core（唯一正式計算入口） |
| 3 | **Negotiation Strategy 整合推演** | 整合成功機率多少？先談誰？ | Decision Engine（EV/verdict）；「先談誰」＝M6 Strategy Engine |
| ✓ | **Decision Report 決策報告** | 下一步該做什麼？ | 組合三引擎輸出（不計算） |

## 跟一般都更試算工具差在哪（護城河）

1. **別人算財務，我們走決策**——試算工具給你一個數字；BUILDER 給你 GO／CAUTION／STOP、
   破局引爆點在誰、退場訊號亮不亮，而且每個判讀都能以 input_hash 溯源回計算來源。
2. **規劃真正驅動人心**——改公設比、改坪型、改車位，逐戶的分回與意願跟著動。
   「你能不能設計一個讓大家都願意的產品」——那才是整合人真正的工作，
   市面上沒有任何工具在做這件事。
3. **從真實案件校準的決策框架**——不是通用財務模型，是用真實都更案逐項比對過的
   容積／坪效／共負引擎，加上把破局模式（內爆／背信／攔胡）寫成回歸測試的決策層。
4. **可稽核的誠實**——UI 零推論、缺欄不臆造、未校準的假設標在結論旁邊，
   跟結論一樣顯眼。工具可以被信任，是因為它先誠實。

## 兩層 IA

```
Workspace（多案）＝案件組合：挑案/匯入/開案
   └─▶ Case Workspace（單案）＝該案容器：STEP 1 → 2 → 3 → ✓ Decision Report
```

## 三條語言紀律

1. **問句屬於卡片，答案屬於引擎**——UI 提出問題並呈現引擎答案（input_hash 溯源），永不自己算。
2. **對外講決策，對內講模組**——行銷/文件/首頁用產品語言；工程檔名不動（映射層）。
3. **Decision Report 是 payoff**——組合 Core＋Decision Engine（＋M6 Strategy）輸出成
   「開發商下一步」，不是遊戲結束畫面。
