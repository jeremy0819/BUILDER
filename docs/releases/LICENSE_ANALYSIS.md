# LICENSE 分析（M1 Task 4）——分析用，**不建立 LICENSE 檔**

> 授權是法律決定，最終由 repo 擁有者拍板。本檔只做選項分析與建議，不代為建立。

---

## 1. 專案性質（影響選擇的關鍵事實）

- 單一開發者（建築/都更 PM），目前為**內部前期評估工具**，非對外商品。
- 內含 **know-how 資產**：L6 費率經真實案校準（護城河）；雖然校準紀錄不進版控，
  但公式結構本身有商業價值。
- 依賴皆為寬鬆授權（Streamlit=Apache-2.0、pandas=BSD、plotly=MIT、openpyxl=MIT、
  jsonschema=MIT）——**不構成 copyleft 傳染**，故下游任何選擇都相容。
- 未來規劃含對外服務（P3 FastAPI/Agent），屆時授權立場會影響能否商業化。

## 2. 選項比較

| 授權 | 適合情境 | 對本專案的利弊 |
|---|---|---|
| **Proprietary（專有／全保留）** | 內部工具、不打算開源、保留商業化彈性 | ✅ 最大化保護校準 know-how 與未來商業化；✅ 不承諾任何人任何事；⚠️ 需在 repo 明示「All rights reserved」與使用範圍 |
| **MIT** | 想開源、要最大採用率、不在意他人閉源再利用 | ✅ 最簡單通用；❌ 任何人可拿去商業化、閉源再包裝，護城河公式等於送出 |
| **Apache-2.0** | 開源＋要專利保護條款 | ✅ 比 MIT 多專利授權與貢獻者條款；❌ 同樣允許他人商業化 |
| **GPL-3.0（copyleft）** | 想開源且要求衍生物也開源 | ✅ 阻止他人閉源再利用；❌ 對「未來想做商業 SaaS」是自綁手腳；❌ 對純內部工具無意義 |

## 3. 建議（供拍板）

**現階段（內部工具、含未變現 know-how、商業化路徑未定）→ 建議 `Proprietary`。**

理由：
1. 保留全部未來選項——之後想開源隨時可從專有「放寬」到 MIT/Apache；反向（開源收回）
   幾乎不可能。**專有是可逆的起點，開源是不可逆的起點。**
2. 校準費率與 L6 結構是核心競爭力，MIT/Apache 等於無償送給潛在競爭者。
3. 依賴授權不強制本專案開源（無 copyleft 傳染），故專有完全合法可行。

**具體做法（待使用者同意後才執行）**：在 repo 根加 `LICENSE` 檔載明
「Copyright (c) 2026 <擁有者>. All rights reserved. 內部使用，未經授權不得重製、散布、商用」，
README 授權段同步標注。若日後決定開源特定部分（如 schema 供互通），可對**個別目錄**
另附寬鬆授權（如 `schemas/` 給 MIT 促進生態），主體維持專有。
