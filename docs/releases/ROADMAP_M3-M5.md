# Roadmap M3–M5 — 案件管理層 → 財務深化 → 法規資料化

> 使用者定案（2026-07-14）：M3 ＝ **Urban Renewal Workflow OS（案件管理層）**，不是新計算工具。
> 本檔給 M3–M5 的里程碑輪廓，並把 M3 的兩條施工線（C 批 Workflow、B 批遊戲）逐批列清。
> 三禁令貫穿全程：**不修改 Core Formula／UI 不自算／不跳過 Schema Version**。
> 版本：M3→`os-v0.3.0`／M4→`os-v0.4.0`／M5→`os-v0.5.0`。

## M4 — Finance Engine 深化

**Why**：M3-A 只做到「均勻分期＝結構示意」；缺 S 曲線、時間價值、貸款攤還。
**禁令**：仍只在 `core/redcf/` 動；校準值（S 曲線權重、費率）🔴 需你核准才落地。

| 批 | 內容 | 驗收 |
|---|---|---|
| M4-1 | `calc_現金流分期` 升級：實案 S 曲線權重（規劃→基礎→結構→裝修→交屋） | 黃金測試鎖關鍵期值；權重值經你核准 |
| M4-2 | `calc_irr()`／`calc_npv()` 建於分期現金流之上 | 黃金測試鎖 IRR/NPV；與 Excel 對照 |
| M4-3 | Loan Engine（建融/土融分期攤還取代單期簡化） | 攤還排程黃金測試 |
| M4-4 | schema **v2.2**（加選填 `result.cashflow`/`irr`/`npv`；走升級流程） | 遷移器＋凍結 hash＋範例可回放 |

**版本**：`os-v0.4.0`。**不做**：Monte Carlo（延後）、AI。

## M5 — Regulation as Data（分縣市法規＋估值係數）

**Why**：law_db 無縣市差異、無生效日；且 M3-B/估值需要「更新前價值係數」——**此處收編你待核准的係數表**。
**禁令**：法規為資料、不是硬寫邏輯；`result.provenance.law_db_version` 帶版本。

| 批 | 內容 | 驗收 |
|---|---|---|
| M5-1 | 拆 `law_db.py` → `regulation/{taipei,newtaipei,taoyuan,…}/`，每條帶條文/上限/來源/生效日 | 切換縣市→容獎上限隨之變；各縣市黃金案 |
| M5-2 | **更新前價值係數落地**（路寬/分區/建物型態）——你核准的對照表進 `valuation.py`＋黃金測試 | 係數表經你核准；黃金值鎖定 |
| M5-3 | `result.provenance.law_db_version` 版本化；schema **v2.3** 帶縣市/生效日溯源 | 舊案重算用對版本法規；遷移器＋凍結 |

**版本**：`os-v0.5.0`。**不做**：AI。
> 🔴 **待你提供**：M5-2 的三張係數表（格式見 `docs/methodology/.../2026-07-13-m3b-street-fabric.md §0`）。
> 在此之前 `calc_更新前價值` 不變、遊戲/儀表板照常。
