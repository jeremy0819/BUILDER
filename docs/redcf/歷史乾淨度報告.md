### ➜ 建議合併策略：**乾淨快照（clean snapshot）＋舊庫轉私有封存**

理由：
1. 歷史中的真實資料不只在檔案 diff 裡，也在 **commit 訊息**裡（例：某 commit 標題直接含
   真實段名與「實際發包數據」字樣、內文含具體金額）。要洗乾淨必須全歷史改寫
   （filter-repo），改寫後的歷史對追溯已無意義，不如快照。
2. 舊庫另有價值：它是「真實案校準」的私有底稿（v4.8 校準 commit 內含實際數字比對），
   **轉私有封存後仍可查**，比洗掉更有用。
3. 快照起點即本報告對應的 tag `v0.2.0-premerge`，工作區已通過 P0-1 驗收。

## 已知缺口（如實揭露，供合併 session 評估）

- **合約無逐層資料**：`project_schema.json` 的輸入區塊（land/building/finance）**不含
  逐層樓板表**，故「從合約 JSON 重算 result」目前不可行（少了 §162 逐層查核的輸入）。
  `min_example.py` 因此讀的是內部格式 `min_input.json`。這是 schema 2.0 的候選項目，
  已記 backlog、**不在凍結期間處理**。
- **LICENSE 不存在**：本庫目前無 LICENSE 檔。授權是法律決定，留給 repo 擁有者：
  內部工具→建議明示 proprietary；要開源→建議 MIT。**合併庫只能有一個授權立場**（P2）。
