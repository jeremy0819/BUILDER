## 為什麼是這份清單、不是直接談合併

上一輪 Urban-Renewal 的介面對齊回覆備忘錄裡，你們自己列了「收到 owners[] 後，我方立即做」
三項，以及「需要貴方提供的」四項。我方在 v4.9 已經把你們要的四項都交付了
（`warnings[]`／`owners[]` 規格／`computed_at`／`core_version`，見上方對照文件）。

**現在球在你們那邊**：規格拿到了，但拿到規格不等於接上了。這份清單就是「規格到位後，
你們那份 repo 實際要做的事」，做完這些，雙方才是站在同一個基準點上談合併，而不是我方
講我方的、你們講你們的。

## P0（優先，建議先做——不需要等我方任何東西）

### 1. 換掉自訂的共負比健檢邏輯，改讀 `result.warnings[]`
你們備忘錄裡提到消費端有一條自判規則：「共負比…（含 **< 70% 具 100% 融資潛力**閘門）」。
這正是我方 v1.1 交付 `warnings[]` 想取代的東西——**現在請把這條自訂規則退場**，改吃
`SHARED_COST_HIGH` / `SHARED_COST_LOW` 這兩個 code。

⚠️ 請注意門檻**不是同一組數字**：我方的門檻是依案件類型＋投報模式動態對照
`law_db.COMMON_BURDEN_RANGES`（危老 25–45%／60% 警示；都更全案管理 30–50%／65% 警示；
都更合建 35–55%／65% 警示），不是你們原本的單一 70% 閘門。這代表換過去之後，同一個案子
在 UI 上顯示的紅黃燈**可能會跟現在不一樣**——這是預期中的行為（因為我方的門檻是拿真實案
校準過的，你們的 70% 是通用估計值），不是 bug，換之前跟內部使用者說一聲。

### 2. 驗證器升級到 schema v1.1，並決定舊格式檔案怎麼處理
`result` 底下 `warnings`／`computed_at`／`core_version` 這三個欄位在 v1.1 是 **required**。
你們目前的驗證邏輯是「缺 `schema_version`／`project.name`／`project.renewal_type`／
`result` 六個必要數值欄 → 報錯」——這條規則沒有涵蓋新的三個必要欄位，**現在照 v1.0 驗證
邏輯的話，v1.1 的檔案不會報錯，但也不會強制要求那三個新欄位存在**，等於驗證器沒真的
升級。請對照最新 `schemas/project_schema.json` 更新驗證規則。

同時請決定：使用者手上如果還有 v1.0 時期匯出的舊檔案（沒有 `warnings`/`computed_at`/
`core_version`），匯入時要怎麼處理？建議：缺這三欄不擋匯入（本來就是可選的向下相容判斷），
但要在 UI 提示「此檔案為舊格式，未含健檢判斷，建議重新匯出」。

### 3. 拿正式範例檔重新壓測，不要只測自己手刻的合成資料
`schemas/examples/` 這次交付了 4 個檔案，**其中 3 個是無 owners、1 個是有 owners（48 戶）**，
且**都內建至少 1 個 warning**（`合成案例D_危老合建.json` 甚至同時觸發 3 個 warning，
可以測多筆 warnings 並列顯示）：

| 檔案 | 用途 |
|---|---|
| `合成案例D_危老合建.json` | EFFICIENCY_OUT_OF_BAND + VOLUME_EXCEEDED + SHARED_COST_HIGH 同時出現 |
| `合成案例A_都更全案管理.json` | 容積剛好打滿的邊界情況（`remaining_floor_area` 極小負值） |
| `合成案例C_防災都更_容積超出.json` | 另一組容積超出＋銷坪比異常組合 |
| `合成範例_owners示範.json` | 48 戶 owners[]，Σ 一致性自檢**乾淨通過**（示範正常情況） |

你們之前用自己寫的合成範例測試，跟我方實際產出的格式可能有欄位順序、四捨五入位數、
巢狀結構等細節落差。**請直接拿這 4 個檔案跑一次匯入 + 健檢 + 逐戶分回（第 4 個檔案），
確認跟你們既有的合成測資結果一致。**

### 4. owners[] 一致性 warning 的 UI 處理
`OWNERS_SHARE_MISMATCH`（Σ land_share 偏離 1.0）與 `OWNERS_VALUE_MISMATCH`（Σ pre_value
偏離 `result.pre_renewal_value`）這兩個 code 目前**沒有在你們原本規劃的健檢清單裡**
（你們備忘錄只提到銷坪比／容積／共負比／§162）。這兩個是新增的，請確認逐戶分回表在遇到
這兩個 warning 時的行為——建議：畫面上還是顯示逐戶分回（不要因為加總對不準就整個不顯示），
但要有明顯提示「清冊加總與 Core 總量不符，數字僅供參考」。

### 6. Phase 2（雙向）的技術棧準備
你們目前是「零依賴純前端靜態站，無 build step、無 Node/Python runtime、無伺服器」。
Phase 2 雙向資料流需要**呼叫我方未來的 FastAPI 端點**（ROADMAP v7）。純前端呼叫外部 API
會有 CORS、金鑰／認證、環境變數管理等現在完全不用煩惱的問題。**建議現在就開始想**：
Phase 2 上線時，你們是要維持純前端＋直接呼叫我方 API，還是屆時要加一層輕量後端代理？
這個架構決策現在想清楚，比 Phase 2 火燒眉毛時才決定好。

### 7. 產出你們那份「合併計畫角色定位文件」

至少涵蓋：你們現況（技術棧、部署方式、目前功能清單）、你們在合併計畫中認為自己該扮演的
角色、你們的邊界（哪些事你們覺得不該做，應該推給 Core 端或第三方）、你們對合併架構的立場

但提前告知你們這份文件的期望內容，可以先開始準備。

## P2（低優先，錦上添花，不影響合併規劃時程）

### 8. 若要加速溝通：開放 repo 讀取權限
目前雙方溝通完全靠手動交換 MD／JSON 檔案。如果 Urban-Renewal 的 repo 是 public（或願意
給我方 read-only access），我方可以直接讀程式碼比對介面實作細節，減少「我猜你們大概是
這樣做」的溝通成本。**這是選配，不做也不影響現有流程**，純粹是效率選項。

### 9. 版本協調流程明文化
目前的協調方式是「schema 升版 → 我方寫備忘錄通知 → 你們照著改」，這次跑起來沒問題，
但沒有寫成正式流程。等雙方都覺得這個節奏穩定了（可能要再跑個一兩輪），可以考慮寫一份
簡短的「版本協調 SOP」，界定：多久算一個週期、誰觸發下一輪討論、破壞性變更的通知前置期。
**現階段不急，先不要為了流程而流程。**
