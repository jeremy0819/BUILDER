## 技術棧與指令

| 套件 | 用途 |
|------|------|
| Streamlit ≥1.30 / pandas ≥2.0 / plotly ≥5.18 / openpyxl ≥3.0 | UI／表格解析／圖表／Excel（版本以 requirements.txt 釘死值為準，Python 3.11） |

```bash
pip install -r requirements.txt
streamlit run app.py        # Web UI
pytest                      # 黃金測試（改公式必跑；test_golden.py）
python make_template.py     # 重產 Excel 對照範本（改 L6 函式後必跑）
python min_example.py       # 最小可跑範例（讀 JSON → 算 → 出 result）
```

## 程式撰寫規範

- 計算邏輯只進 `core/`（不碰 Streamlit）；app.py 純 UI 零公式
- 變數用中文，函式名中英混用（`calc_坪效`）；對外 JSON key 英文（分界＝core/contract.py）
- 每個計算區塊加中文註解，標層級（L2/L3…）與法規條文
- 狀態色：超出 `st.error`／接近 `st.warning`／合規 `st.success`
- 小數 2 位；面積 m²、坪數「坪」、金額「萬元」
- 絕不寫死結果，全部從 L1 輸入推導；新增計算函式必加測試

## Git 工作流程

- 分支：以**當前 session 被指定的分支**為準（舊文件中的 `claude/claude-md-docs-ls9Bu` 已過時）。
- commit 前必過：`pytest` 全綠＋真實段名 grep 零命中（指令見 CLAUDE.md 紅線）。
- 只 `git add <改過的檔>`，不用 `-A`；訊息 `feat:/fix:/docs:/refactor:` 開頭。
- 上線：merge 到 `main`（`--no-ff`）→ Streamlit Cloud 自動部署。單人專案正常開發直接 push，
  需外部審查（建築師確認計算邏輯）才開 PR。

## 版本號更新規則

改 `app.py` 時同步更新：頁尾 `_build = "YYYY-MM-DD"`、頁尾與 Hero 的 `RE-DCF-Tool v4.X`（+0.1）。
（技術債：版號散落 4 處，合併搬遷時抽常數——見 BUILDER/docs/architecture/MIGRATION_PLAN.md 步驟 2。）
`CORE_VERSION`（core/_version.py）獨立於 UI 版號，是消費端追溯依據，只在公式/費率變動時 bump。


## UI 細節

- 四角色快覽卡：🏗️建管（容積合規）／📐建築師（免計查核）／📋地政士（容積帳）／💰開發商（投報）
- 營造坪基準切換：「前期保守（銷售坪）」vs「實務（允建坪）」——用銷售坪當營造基準會高估成本達 60%
- 逐層表格：`st.data_editor`，可上傳 xlsx/csv（`解析上傳` 彈性欄名）或帶入合成案例 A–D

## Excel 對照範本

`都更全案投報_對照範本.xlsx`（make_template.py 產生）：Sheet1 坪效及獲利分析（黃=輸入、
藍=關鍵指標，與 `calc_投報全案` 1:1）；Sheet2 參數來源與說明。**改 L6 函式後必重跑同步**。

## 建築師反饋收集協議（說「有出入」時先問三題再改 code）

```
Q1 卡在哪一步？（找不到輸入欄位／數字不對／格式看不懂）
Q2 實際操作是什麼？（Excel 匯入還是手動？他慣用的欄位名稱：梯廳還是走廊？陽台還是露台？）
Q3 他期望看到什麼 vs 工具顯示什麼？（兩個數字或截圖並列）
```

## 敏感資料

- `供py/`：真實案件 Excel，.gitignore 排除，**絕對不可 commit**
- 真實案名/段名/金額/姓名不進版控（含檔名與 commit 訊息）
- 校準紀錄只進私有紀錄（建議建私有 calibration repo，見 BUILDER governance 交接信）
