# 兩庫合併前置：RE-DCF 側任務清單

> **文件用途**：交給 RE-DCF-Tool 側（session）執行的**合併前置任務**。逐項附驗收標準，未來執行合併的新 session 將照此驗收。
> **對讀文件**：`docs/合併計畫-Urban-Renewal側角色說明.md`（我方交接文件）
> **原則**：合併是**搬家不是改建**——本清單只要求「可安全搬、搬得動、搬完可驗」，不要求任何新功能。
> **日期**：2026-07

## P0 — 不完成就不合併（阻斷項）

### P0-1 範例檔去識別化（最重要）
`schemas/examples/` 現有範例以**真實段名**命名（原名已於 P0-1 完成去識別化，本文件不再複載），且自述「用真實在建工程數據跑出來」。合併後這些會**永久進入新庫的 git 歷史**，事後洗不掉。

- [ ] 檔名改為合成代號（如 `合成案例A_危老合建.json`），**檔名與 JSON 內 `project.name` 都要改**
- [ ] 內容數字**擾動或重生**：不得是真實案件原始數據的直接匯出——可用 `generate_examples.py` 以「校準後公式＋虛構輸入」重跑，保留「會觸發哪些 warnings」的測試性質即可
- [ ] `generate_examples.py`、測試碼、docs、README 內文同步去除真實段名字串
- **驗收**：真實段名檢查（現收斂為 `check_no_real_names.sh`，含檔名）零命中；範例仍能觸發原設計的 warning 組合

### P0-2 全庫敏感資料掃描＋git 歷史盤點
- [ ] 掃描工作區：真實案名／姓名／電話／地號／實際發包金額，出現在程式註解、docs、測試、Excel 對照範本者一律去除
- [ ] 掃描**git 歷史**（`git log -p` 全文）：列出歷史中是否存在過真實資料的提交
- [ ] 產出一頁「歷史乾淨度報告」：**乾淨 → 合併可用 subtree 保留歷史；不乾淨 → 改用乾淨快照＋舊庫轉私有封存**。這個結論直接決定合併策略，請明確寫出建議
- **驗收**：報告存在且結論明確（乾淨／不乾淨＋建議策略）；工作區掃描零命中

### P0-3 RE-DCF 側交接文件
- [ ] 比照我方格式寫一份《合併計畫-RE-DCF側角色說明.md》，至少含四塊：**資產清單**（模組、黃金測試、契約、範例產生器）／**紅線**（不可被合併破壞的事）／**已知風險與立場**／**你們自己的技術債清單**（老實列，例如 owners 輸入 UI 未做、IRR/NPV 未做）
- **驗收**：新 session 能只靠兩份交接文件對讀就開工，不需回頭問基本問題

### P0-4 main 分支收斂
- [ ] 所有要帶進合併的工作全數合入 `main`，工作區乾淨、無散落分支
- [ ] 為當前狀態打 tag（如 `v0.2.0-premerge`），作為合併基準點
- **驗收**：`git status` 乾淨；tag 存在；`main` 可從乾淨 clone 直接跑起來

## P1 — 可攜性（決定搬過去會不會壞）

### P1-1 core/ 可獨立匯入驗證
你們自述「計算核心已模組化、UI 完全分離」——合併後 core 會搬到 `core/redcf/`，請先自證：

- [ ] 在乾淨環境（無 Streamlit）`import core` 並跑一筆完整計算成功——證明 core **零 UI 依賴**
- [ ] core 內部無寫死的絕對路徑／工作目錄假設（搬目錄不會斷）
- **驗收**：提供一段 10 行內的最小可跑範例（讀合成 JSON → 算 → 出 result），新 session 在 `core/redcf/` 新路徑下重跑通過

### P1-2 黃金測試一鍵化
- [ ] 全部黃金測試收斂為單一指令（如 `pytest`），乾淨 clone 直接過
- [ ] 測試中引用的對照數據（Excel 對照等）同步確認去識別化（呼應 P0-2）
- **驗收**：`git clone → pip install -r requirements.txt → pytest` 三步全綠

### P1-3 相依與啟動路徑文件化
- [ ] `requirements.txt`（或 pyproject）版本釘死；註明 Python 版本
- [ ] README 寫清楚 **Streamlit 啟動指令**與埠號假設——合併後新庫 README 要並列「靜態站（Pages）」與「計算工具（Streamlit）」兩條啟動路徑，你們這半邊的說明由你們提供
- **驗收**：照 README 從零啟動 Streamlit 成功

### P1-4 契約凍結宣告
- [ ] `project_schema.json` 1.1 宣告凍結：合併期間**不做任何 schema 變更**（含非破壞性新增），有需求先記到 backlog，合併完成後再依既定規矩（知會→bump）處理
- **驗收**：合併期間 schema 檔 hash 不變
