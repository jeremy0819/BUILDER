# M3-B 規格：基地文理（Street Fabric）— 參數化街區・三態地主・選屋籌碼

> 使用者需求定案（2026-07-13）：①案件參數輸入要與評估儀表板同一套格式；②三態地主與
> 選屋籌碼的隨機性要源自「基地的原始文理」；③地主數量可選擇/手動調整；④人格之間有
> 關聯／家族性的宇宙連動。本規格把需求轉成可分批施工的合約——**M3 的開始大工程**。
> 權威順序：本規格 < ARCHITECTURE.md/governance（衝突時以凍結架構為準）。

## 0. 兩層輸入架構（回答「係數在哪裡輸入」）

都更評估的輸入分兩層，**不能混**：

| 層 | 內容 | 誰輸入・多常改 | 落點 |
|---|---|---|---|
| **A. 案件事實**（每案不同） | 基地面積、容積率、開發模式、路寬、使用分區、屋齡、建物型態、地主數量、清冊… | 使用者**每案／每局**輸入 | schema `input.*`（Streamlit 表單、遊戲開局、儀表板匯入——**同一份合約格式**） |
| **B. 校準係數**（全 OS 共用規則） | 「路寬 X 米 → 更新前價值加成多少」這類**對照表** | repo 擁有者**核准一次**，之後只在法規/市場變動時改（🔴 紅線） | `core/redcf/valuation.py` 係數表＋黃金測試 |

所以：**你不需要每局輸入係數**——每局輸入的是路寬/分區/型態這些「事實」（A 層），
Core 拿事實去查你核准過的係數表（B 層）。這正好和你的直覺一致：遊戲開局輸入基地
參數＝A 層＝與儀表板格式一致（同一份 schema v2.1 合約）。

### B 層係數表——待你填值核准的格式（填好我就實作進 Core＋黃金測試）

```
路寬係數（面前道路寬度 → 更新前土地價值乘數）
  <8m：____   8–15m：____   15–25m：____   >25m：____
分區係數（使用分區 → 乘數）
  住3：____   住4：____   商2：____   商3：____   （需要的分區自行增列）
建物型態係數（既有建物 → 建物現值乘數）
  RC：____   加強磚造：____   磚木造：____   （另：屋齡折舊曲線是否沿用現行？）
```
> 在你核准前，`calc_更新前價值` 維持現狀（不臆造）；遊戲/儀表板照常運作。

## 1. 目標圖像（一句話）

**同一份案件輸入（A 層），儀表板拿來評估、遊戲拿來開局**——基地文理（面積/容積/
模式/地主數/清冊）決定財務（Core 算）也決定街區（誰住在這、誰跟誰是家族、誰是
關鍵戶），評估與遊戲徹底同宇宙。

## 2. 自訂案件開局（與儀表板一致的輸入）

- **主路徑（SSOT-safe）**：Streamlit 填案件參數＋owners CSV → 匯出 schema v2.1 JSON
  → 遊戲 PLAN PHASE 新增「📂 匯入案件」→ 讀 `result.*`（允建/共負/分回/投報/warnings）
  當本局權威財務、讀 `input.owners[]` 生成街區。**遊戲端零計算**——靜態頁不重算，
  只消費 Core 匯出（跟 SCEN 同一紅線）。
- **快速路徑**：無 JSON 時沿用現行 6 內建方案（SCEN）。
- **難度映射（遊戲規則表，非財務公式，寫死並測試）**：
  `共負比 <0.65 → 開局信任 +4｜0.65–0.70 → 0｜>0.70 → −5`；
  `容積超出（remaining<0）→ 事件「設計變更」風險 +`；warnings 數量 → 輿論事件頻率。

## 3. 街區文理生成（參數化 SIMCORE）

| 參數 | 選項 | 預設 |
|---|---|---|
| 地主數量 N | 24 / 36 / 48 / 60（或手動 12–80） | 48 |
| 開局同意比例 | 50–85%（手動） | 70%（→48 戶＝34） |
| 家族密度 | 低/中/高（家族戶佔比 ~10/20/30%） | 中 |
| 關鍵戶數 | ⌈N/48⌉ 檔位（48 戶=1、60 戶=2） | 1 |
| 文理種子 seed | 匯入案件＝`input_hash` 前 8 hex；內建案＝固定 | 固定（可重現） |

- **決定性**：同 seed 同街區（mulberry32；headless 可測）。
- **家族宇宙連動**：家族＝**鄰接格聚落**（2–4 戶連通區塊），由 seed 生成；同家族
  共享「家族性格底色」（例：守成/投資/漂泊），個人性格在底色上變異——一人翻轉
  → 全家 +8 的既有機制不變，但家族成員與位置不再寫死。
- **人格 pool 依文理加權**：商業區文理→店主/投資客多；老住宅文理→長者/祖厝多
  （加權表＝遊戲規則，非計算）。
- 格盤尺寸表：24→6×4、36→6×6、48→8×6、60→10×6。

## 4. 三態地主與選屋籌碼（機制）

狀態機（對齊《整合人手冊》§6 與 `owners.min_unit_eligible`）：

```
未同意 ──說服──▶ 同意（未選屋） ──發「選屋順位」──▶ 同意且選屋 ✅（鎖定）
   │                    │（拖到結算）
   ▼                    ▼
 不同意（送件後）→ 領補償金        公開抽籤（軟懲罰：抽到差戶別，信任事件）
```

- **選屋順位＝新籌碼**：每週有限張「順位券」，發給誰＝玩家決策；早順位＝好戶別
  ＝該戶信任↑且**鎖定**（後續負面事件免疫）——把「選屋順序本身是可分配的利益」
  做成玩法。
- **抽籤軟懲罰**：結算時「同意未選屋」者公開抽籤；抽到差戶別→敘事事件（不追溯
  改變勝負，是結局色彩）。
- **結局＝接 M3 Core 成果**：結算畫面直接渲染 `result.owner_allocations[]` 逐戶
  分回表＋找補（權威數字），三態各自標註（選屋/抽籤/補償金）——玩家第一次在
  遊戲裡看到「每一戶最後拿到什麼」，全部 Core 產出。

## 5. 施工順序（每批獨立可驗收，不綠不進下一批）

| 批 | 內容 | 驗收 |
|---|---|---|
| B1 | SIMCORE 參數化（N/同意比/seed；家族・關鍵戶・人格改程序生成） | headless 測試矩陣改 N∈{24,48}：決定性/家族連通/可贏帶/掛機必輸 全綠 |
| B2 | PLAN PHASE「基地文理」步驟（N、同意比、家族密度、seed UI） | 實開：不同參數→不同街區、同 seed 可重現 |
| B3 | 三態＋選屋順位券＋抽籤事件 | 新增不變式測試：三態守恆 Σ=N、順位券效果、結算正確 |
| B4 | 匯入案件 JSON（v2.1）開局＋難度映射表 | 匯入範例 v2_1\_案例D → 財務=result、街區=owners、零 console error |
| B5 | 結局權變表（owner_allocations 渲染） | 結算數字與 JSON 逐戶一致（只讀） |

**紅線不變**：遊戲層零財務公式；SIMCORE 區塊改動必跑 headless；改 Core 必跑 pytest；
B 層係數未核准前不進 Core。V4 simulator.html 封版不動。
