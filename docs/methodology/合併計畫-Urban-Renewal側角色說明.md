# 兩庫合併計畫：Urban-Renewal 側角色說明（交接文件）

> **文件用途**：RE-DCF-Tool（坪效／投報計算）與 Urban-Renewal（都更方法論／互動工具）兩 repo 合併計畫之**我方交接文件**。本文件將與 RE-DCF 側的對應文件一起，交給執行合併的**新 session** 對讀後開工。
> **寫作前提**：讀者（新 session）對兩專案**零上下文**，本文件自足。
> **日期**：2026-07　**我方現況版本**：Simulator V4 封版／evaluator 1.2（支援 Core schema 1.1）

## 一、我是誰：Urban-Renewal 側的角色定位

合併後的目標形態是 **Urban Renewal OS**（架構備忘已定：Core 計算 → Data 契約 → Dashboard 管理 → AI 智慧層）。在這個分工裡：

| 角色 | 我方（Urban-Renewal） | 對方（RE-DCF Core） |
|---|---|---|
| **定位** | **展示層＋知識層**：儀表板、沙盤、教學試算、方法論文件 | **計算核心（SSOT）**：容積／坪效／共負／估值公式唯一來源 |
| **技術棧** | 純前端靜態站（單一自含 HTML＋vanilla JS，零依賴、零建置，GitHub Pages 部署） | Python／Streamlit，`core/` package 已模組化 |
| **對數字的權限** | **只讀、只顯示、不重算**（SSOT 紅線） | 唯一可產生數值與權威警告（`warnings[]`）者 |
| **合併中承擔** | 見下方第三節六項職責 | 計算引擎遷移、公式與黃金測試、契約維護 |

**一句話**：我是「把 Core 算出來的東西讓人看懂、學會、管得動」的那一側；合併時我同時擔任**介面契約守門人**與**資料紀律稽核**。

## 三、合併過程中我方的六項職責

1. **repo 結構共同定案**：以 OS 架構備忘為基礎提案——
   ```
   Urban-Renewal-OS/
     apps/web/        ← 我方 5 個靜態頁（Pages 部署根）
     core/redcf/      ← 對方計算引擎（Python）
     schemas/         ← 共用契約（project_schema.json，唯一資料合約）
     docs/            ← 兩庫方法論文件整併
     tools/           ← 產生器 py、headless 測試
   ```
   關鍵原則：**靜態與 Python 共存但不互相污染**——Pages 只服務 `apps/web`，Streamlit 獨立啟動，兩者只透過 `schemas/` 的 JSON 對話。
2. **介面契約守門**：合併「不因為住在同一個 repo 就繞過契約」。schema 1.1 仍是唯一介面；任何破壞性變更照既定規矩（先知會、bump version）。我方持續維護消費端驗證器。
3. **靜態頁遷移與部署**：搬遷 5 頁與相對路徑修正、`.nojekyll`、Pages 設定驗證；遷移後跑完整 headless 迴歸。
4. **文件整併**：兩庫 README／docs 合併去重（我方管方法論與使用者文件，對方管公式與黃金測試文件），路線圖 V5–V7 掛到新 repo。
5. **資料紀律稽核（合併時最重要的一刀）**：合併前逐檔掃描**兩庫**——特別提醒：對方 `schemas/examples/` 現有範例檔以**真實段名**命名（原名已去識別化，此處不複載）且自述源自真實案例校準，**合併前必須去識別化或改名**，否則真實案件線索會永久進入合併庫的 git 歷史。我方負責把關、開清單、驗收。
6. **git 歷史處理建議**：優先 `git subtree`（或 merge --allow-unrelated-histories）**保留兩庫歷史**；但若對方歷史中含未去識別化的提交，改用「乾淨快照＋舊庫封存唯讀」策略——這個判斷以第 5 點稽核結果為準。

## 四、我方紅線（合併不可破壞）

1. **SSOT**：Dashboard／沙盤／試算永遠不寫計算公式，數字只來自 Core。
2. **靜態純度**：`apps/web` 到 V6 為止零依賴、零建置、可離線——不因合併引入 bundler／framework。
3. **零真實案件資料**：真實案名、地號、金額、姓名不進版控（含範例檔名）。
4. **V4 封版**：simulator 不因合併重寫；改動走 V5 路線圖（沙盤劇本橋接等）。
5. **95 分原則**：合併是**搬家不是改建**——先原樣遷移＋全綠驗收，才開始任何新功能。

## 五、已知風險與我的預先答案

| 風險 | 我的立場 |
|---|---|
| 技術棧異質（靜態 vs Streamlit） | 不硬融合。monorepo 共存、JSON 契約對話；Pages 服務靜態、Streamlit 另行啟動（README 寫清楚兩條啟動路徑） |
| 對方範例檔含真實段名 | 合併前置條件（見三-5），不解決不合併 |
| 兩庫 docs 重疊（如坪效敘述） | 以「公式歸 Core docs、方法論歸我方 docs」切分，重複處由我方版本收斂（我方已維持嚴格的 KNOWN/FRAME 標記紀律） |
| 合併後路徑斷鏈（頁面互鏈、README 連結、RE-DCF 外鏈） | 我方遷移職責內；headless 測試加連結檢查 |
| 新 repo 命名與舊庫處置 | 建議 `Urban-Renewal-OS`；舊庫封存唯讀＋README 指向新庫（保留外部已分享連結的可達性） |

## 六、給新 session 的啟動順序（建議）

1. **對讀兩份交接文件**（本文件＋RE-DCF 側文件），列出兩側自述的「紅線」與「資產」，衝突處先問使用者。
2. **資料紀律稽核先行**：掃兩庫所有檔案與檔名（重點：對方 examples 的真實段名），出去識別化清單，完成後才動 git。
3. **提出 repo 結構與 git 歷史策略**（subtree 保史 vs 乾淨快照），給使用者二選一。
4. **原樣遷移**：先搬不改，Pages＋Streamlit 兩條啟動路徑都跑通、headless 迴歸全綠。
5. **整併 README／docs**、掛上 V5–V7 路線圖，宣告合併完成。
6. **之後才**依路線圖開 V5（沙盤劇本橋接等）。

> ⚠️ 提醒新 session：兩份交接文件都是**各自陣營的自述**，對讀時注意雙方對同一件事的描述差異（例如共負比警示門檻、examples 資料來源），差異即風險，先釐清再動手。
