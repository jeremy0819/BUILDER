# 給未來 session（與使用者）的信

> 寫於 2026-07 Architecture Freeze session（Fable 5）。這封信講三件事：沒被問到但最重要的事、
> 這套制度會怎麼死、以及我對自己哪些產出信心最低。

## 一、三件沒被問到、但我認為最重要的事

### 1. 最急的不是架構，是舊部署與舊庫的下架動作
乾淨度報告寫得很清楚：舊 Streamlit URL 曾公開顯示真實案名，兩庫 git 歷史含真實金額。
架構文件寫得再好，這件事沒做完，風險每天都在。**這是使用者本人要動手的（部署後台＋repo
設定），任何 session 都代辦不了。** 建議：下次打開 GitHub 的第一件事就是它（MIGRATION 步驟 4）。

### 2. 校準紀錄是全專案最有價值的資產，但它沒有制度化的家
公式誰都寫得出來，「±5% 內對得上真實案」的費率校準才是護城河。目前校準紀錄「不進版控、
存私有紀錄」——但私有紀錄具體在哪？若只散在對話紀錄或某台電腦，等於沒有。建議：開一個
**私有** repo（如 `urban-renewal-calibration`），每次校準一份 md（日期/案件代號/對照結果/
調整了哪個費率/新 core_version），公開庫的 CHANGELOG 只寫「經真實案校準，紀錄見私有庫」。

### 3. 數字的品味驗證，模型永遠代替不了你
弱模型會算、會測、會比對，但「這個共負比看起來不對勁」的直覺只有你有。建議一條個人紀律：
**每次 core_version 變動後，親自開一次工具跑案例 C，肉眼看 L2→L6 的每個數字**（30 秒）。
黃金測試防迴歸，你的眼睛防「測試本身鎖錯值」。同理：法規改版（容獎、§162）只有你會在
業界第一時間聽到——聽到就開 issue，別依賴模型「知道」。

## 二、這套制度最可能的退化方式（與預防）

1. **文件長回去**：每個 session 都想「補充一點」，CLAUDE.md 半年後又是 337 行。
   → 預防：MAINTENANCE §3 的行數上限是硬規則；審查型發包時把「行數超標」列為 finding。
2. **「就這一次」的侵蝕**：趕著看結果，跳過 pytest／繞過 schema 凍結／「先把公式放前端等 API 好了再搬」。
   → 預防：紅線五條（MODEL_DISPATCH §6）不可談判；弱模型遇到「想例外」＝R3 停下問使用者。
3. **governance 與現實脫節**：模型 ID 過期、路徑改了文件沒改，弱模型照舊文件行動出錯，
   然後開始**不信任整套文件**——這是最致命的，制度失去公信力就全垮。
   → 預防：發現任何一處過時，當場修＋記 LESSONS（比寫新內容優先）；每季（或每次大版本）
   派一次 read-back 驗證 governance 全部路徑與指令仍有效。
4. **派工制度退化成獨角戲**：弱模型傾向自己動手不發包，context 悄悄爆掉、品質悄悄下滑。
   → 預防：CLAUDE.md 索引第一條就指向 MODEL_DISPATCH；「>400 行不整讀」是機械規則，好遵守。

## 三、誠實條款：我信心最低的產出

1. **D1 monorepo 裁決（信心中）**——它與 RE-DCF ROADMAP 原則 6、以及使用者某次貼文
   「Repository 架構已定案＝獨立 repo」相抵觸。我依據「最新指示優先＋單人維護的實證痛點」
   裁決，但使用者可能有未說出的理由（例如想保留兩條獨立的部署/展示故事線）。
   回退成本已寫在 ARCH_REVIEW 附錄：否決 D1 不影響 governance/CI/schema 策略。
2. **交叉核對測試的可行性（信心中低）**——教學層的輸入是簡化參數，與 Core 輸入不完全同構，
   可能需要一層映射才能餵同一案例；若實作發現成本過高，降級為「示意標示＋公式凍結」即可，
   不要硬做。這條在 P1 動工時要先花 30 分鐘驗證可行性再排程。
3. **schema v2.0 欄位草案（信心中）**——區塊結構（input/result/provenance）我有把握，
   具體欄位名與巢狀細節是草案級，P1 動工時以 core 實作現況為準微調，不必忠於我的草稿。
4. **模型調度的兩個未確認值（已標註）**——future session 能否 spawn fable、Opus 導流是否
   耗 Fable 額度。都寫了「未確認＋實測方法」，請勿當事實引用。
5. **檔案未推前的狀態**——若你讀到這封信但發現三個 repo 的分支沒有這些檔案，代表當時
   session 在 push 前中斷；以 BUILDER 分支 `claude/urban-renewal-os-arch-freeze-svoum0`
   上實際存在的檔案為準。
